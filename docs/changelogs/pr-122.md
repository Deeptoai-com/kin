# PR #122 — docs(prd): multi-model switching — cross-account, health-gated menu

## TL;DR
- New PRD `docs/project/prd/2026-06-multi-model-switching-prd.md` (233 lines, CN) speccing **v1 multi-model switching across Anthropic-protocol connections/accounts**, gated by a real-time health probe.
- Scope reframed from "same-gateway (ARK) only" → **cross-connection / cross-account**, reusing the existing per-request worker spawn to override `ANTHROPIC_BASE_URL` / auth / `ANTHROPIC_MODEL` per chat.
- Config lives in an **out-of-repo YAML** (`MODELS_CONFIG_PATH`); secrets stay in `secrets.env` — YAML only references env-var names via `tokenEnv`.
- **Menu = `enabled && healthy && in-TTL`**; probe = real `POST {baseUrl}/v1/messages` with `max_tokens:1` (proves reachable + authed + model accepted). `429` counted as healthy.
- Sub-agent / background tier follows the selected connection via per-Connection/Model `aliases` to prevent cross-account leakage.
- `ROADMAP.md` NEXT entry rewritten to point at the PRD; research doc gets a scope-update banner deferring to the PRD on conflict.

## Why
Today the model is a deploy-time singleton: `ws-server.mjs` reads `ANTHROPIC_MODEL` at startup, the composer's "GLM 5.0" badge is cosmetic, and the FE never sends a `model`. Owner steer on 2026-06-07 raised the bar — **any Anthropic-protocol model that is currently usable, including across accounts/gateways, should be switchable, and only when actually reachable**. This PRD captures that scope before any code lands, superseding the earlier "MVP = same-gateway ARK only" framing in the research doc.

## What changed

### Docs (only — no code, no migrations)

**New — `docs/project/prd/2026-06-multi-model-switching-prd.md`** (14 sections):

- **§1 Audit** — model is deploy-time singleton; composer badge is dead; `skillSlug` / `permissionTier` already proves the store → ws-adapter → ws-server → worker plumbing; per-request worker spawn is the key enabler (no SDK 0.3.x needed).
- **§2 Goals / Non-goals** — G1–G5: user-pick per run, cross-connection, health-gated menu, sub-agent consistency, secrets out of repo/FE. Non-goals: failover, OpenAI-only translation, DB-backed registry, capability gating, per-message billing.
- **§3 Data model** — two layers: **Connection** (`id`, `baseUrl`, `authStyle: bearer|x-api-key`, `tokenEnv`, optional `aliases`) and **Model** (`id`, `connection`, `model`, `enabled`, `tags`, optional `aliases`). Runtime `health` / `lastProbeAt` / `probeError` not persisted.
- **§4 Config** — YAML at `MODELS_CONFIG_PATH` (Mac: `~/oxygenie-deploy/models.yaml`; container: `/data/config/models.yaml`); repo ships `models.example.yaml`; secrets resolved from env-var names at boot. DB + admin UI deferred to v2.
- **§5 Health probe** — `POST {baseUrl}/v1/messages` w/ `anthropic-version: 2023-06-01`, body `{model, max_tokens:1, messages:[{role:"user", content:"ping"}]}`. Classified: `200`→healthy; `401/403`→`unhealthy(auth)`; `404/400`→`unhealthy(model)`; DNS→`unhealthy(network)`; timeout→`unhealthy(timeout)`; **`429`→healthy** (warn only). TTL 5–10 min + startup full sweep + manual retest. Explicitly notes the changedoc workflow's `/api/coding/v3` (OpenAI protocol) is **not** the right probe path.
- **§6 Visibility state machine** — table for `enabled × health(in-TTL) → menu state`; selectable iff `enabled && healthy && in-TTL`; default model must be healthy or auto-fallback to first healthy with warn; unhealthy items greyed with reason (`hideUnhealthy` opt-in).
- **§7 Selection → run flow** — `chat-session-store.selectedModelId` → `chat` `InboundMessage` gains `model?: string` → `handleChat` resolves → builds `workerEnv` overriding `ANTHROPIC_BASE_URL` / `ANTHROPIC_API_URL` / auth (mutual-exclusion of `ANTHROPIC_AUTH_TOKEN` vs `ANTHROPIC_API_KEY` to avoid SDK auth confusion) / `ANTHROPIC_MODEL` / `ANTHROPIC_DEFAULT_*` / `CLAUDE_CODE_SUBAGENT_MODEL`. Worker logs `[Worker] Model: <id>`. v1 = store-only; suggests `agent_session.model` column for resume/echo.
- **§8 UX** — grouped dropdown by Connection, health dots, hover-reason for unhealthy, disabled while running; session-info shows current model; optional read-only health panel + "retest all".
- **§9 Security** — tokens never leave backend; FE sees only `id/label/health/tags`; YAML stores env-var names only; keep `ENABLE_STRUCTURED_OUTPUTS=false`; SDK pinned to `0.2.112`.
- **§10 Tests** — unit (YAML parse/validate, state machine, selection validation, `workerEnv` construction incl. bearer↔x-api-key mutual exclusion + alias overrides), integration (real ARK probe, bad-token → `unhealthy(auth)`), manual (cross-account switch verified via worker logs).
- **§11 AC v1** — AC1–AC6: health-gated menu, real per-request routing, isolated token failures, default fallback, token redaction, sub-agent connection consistency.
- **§12 Phasing** — v1 (this) = file config + probe + selection + per-request `workerEnv` + real picker (store-only). v2 = `agent_session.model` persistence + per-message records + DB registry + admin UI (Skills-style). Phase 4 = failover/routing, per-capability key split, capability gating, OpenAI-only translation proxy evaluation.
- **§13 Owner decisions pending** (6) — reject vs silent-fallback on unhealthy/invalid model; grey vs hide unhealthy; probe TTL; config file location; selection granularity (per-session recommended); probe via `/v1/messages` vs SDK micro-query.
- **§14 Implementation checklist** for v1 landing — `models.example.yaml`, `src/config/model-registry.*`, probe module, `ws-server.mjs` `handleChat` changes, `ws-query-worker.mjs`, `src/claude/adapters/ws-adapter.ts` (`chat.model?` + `listModels` / `probeStatus` channel), `chat-session-store.ts` (`selectedModelId`), `chat-composer.tsx` (dropdown), optional `agent_session.model` migration.

**Updated — `docs/project/ROADMAP.md`** (~+8/-6): NEXT "Multi-model selection" entry rewritten to describe **v1 as cross-connection/cross-account** (YAML registry + health probe + per-request worker-env routing + real picker), with DB + admin pushed to v2 and failover/capability-gating to Phase 4. Points at the PRD as spec of record; research doc kept as rationale.
