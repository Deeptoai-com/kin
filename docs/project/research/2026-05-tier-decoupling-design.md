## 0. 一句话目标

让 `ws-server` 从"有状态单点"变成"可水平扩 N 份的无状态网关",使会话状态、工作区、
worker 执行三者都不再被绑死在某一个 node 进程里 —— 这是"撑住成百上千并发"的根本前提
(比执行后端选型更底层)。

## 1. 现状(全部经代码核实,非推测)

`ws-server.mjs` 当前是**有状态单进程**。但好消息是大部分状态其实已经外置了:

| 状态 | 现在在哪 | 是否已可共享 |
|---|---|---|
| 会话元数据(sdkSessionId / claudeHomePath / userId / title) | **Postgres `agentSession` 表**(`getWorkspaceSession` 读、`persistSession` 写) | ✅ 已在 DB |
| 工作区文件 | 本地磁盘 `SESSIONS_ROOT/{userId}/sessions/{sessionId}/workspace`(`CLAUDE_SESSIONS_ROOT` 可配置) | ⚠️ 已路径化,但在本地盘 |
| 会话历史(消息) | CLAUDE_HOME 下的 `.claude/projects/**/*.jsonl`(由 SDK 写) | ⚠️ 在本地盘 |
| `sessionMapping`(workspaceSessionId→sdkSessionId) | **模块级内存 `Map`(line 225)** | ❌ 纯内存缓存 |
| `ws.workerProcess`(运行中的 worker) | **连接对象上,绑定到本 node 进程** | ❌ 进程内 |
| `ws.userId / ws.cookie / ws.sdkSessionId / ws.workspaceSessionId` | 连接对象上 | ✅ 连接级,天然随连接,无需外置 |

**两个真正的"扩展障碍":**

1. **`sessionMapping` 内存缓存**:好在它在 line 1071-1078 **已经有 DB 回退**
   (`getWorkspaceSession` 查 `agentSession.sdkSessionId`)。也就是说**DB 已是真值,这个 Map
   只是缓存** —— 可以安全删除/替换。

2. **`ws.workerProcess` 进程内绑定**:worker 子进程被 `spawn` 在持有该 WS 连接的那个 node
   进程里;abort(line 692-699、1299-1305)靠 `ws.workerProcess.kill()`。**一旦网关有多份,
   连接 A 落在实例 1、它的 worker 也只在实例 1** —— 实例 2 收到针对同会话的请求/abort 时
   够不到那个 worker。这是横向扩展的核心障碍。

**已澄清的误解**:之前概述里提到的 `activeWorkers` / `userSessionLocks` 两个 Map
**在代码里并不存在**(grep 为空)。唯一的模块级会话 Map 就是 `sessionMapping`。

## 2. 目标架构(deer-flow / 12-factor 模式)

```
          ┌─────────── 客户端(浏览器 WS) ───────────┐
          │                                          │
   ┌──────▼──────┐   ┌──────▼──────┐   ┌──────▼──────┐
   │ 无状态网关 1 │   │ 无状态网关 2 │ … │ 无状态网关 N │   ← 可水平扩,任意一份服务任意会话
   └──────┬──────┘   └──────┬──────┘   └──────┬──────┘
          │ 入队/查状态        │                  │
   ┌──────▼──────────────────▼──────────────────▼──────┐
   │  共享状态:Postgres(会话元数据)+ Redis(运行态/缓存) │
   │  任务队列:BullMQ(Redis)                            │
   │  工作区:对象存储 S3/MinIO(或共享卷)                 │
   └──────┬──────────────────────────────────────┬──────┘
   ┌──────▼──────┐                          ┌──────▼──────┐
   │ worker 池 1 │ …                        │ worker 池 M │   ← 从队列取活,独立伸缩
   └─────────────┘                          └─────────────┘
```

要素:
- 网关只做:鉴权、收发 WS、把"跑一条消息"作为任务入队、把任务产出的事件流回客户端。
- worker 池从队列取任务执行(执行后端仍是 PR-1 的 `ExecutionRuntime`,③ 决定具体后端)。
- 网关↔worker 的事件流 + abort 通过共享通道(Redis pub/sub)而非进程内引用。

## 3. 增量 PR 序列(每步可独立验证、可回滚;不一次性重写)

> 原则:每步都是小 PR、有真实运行时验证、可单独回滚。**绝不一次性大爆炸重写。**
> 标 ⚠️ 的是单向门 / 需要你在该步前再确认的。

### PR-A:删除 `sessionMapping` 内存缓存,全部走 DB(纯可逆,零行为变化)
- 把 3 处 `sessionMapping.get/set` 改为 `getWorkspaceSession` DB 查询(line 1071-1078 已示范该模式)。
- DB 已是真值,删缓存后行为不变;唯一代价是每次 resume 多一次 DB 查询(可接受,且本就有回退路径)。
- **验证**:启动 ws-server,真实 resume 一个会话,确认 sdkSessionId 正确解析、对话续上。
- **回滚**:单 PR revert 即可。这是整个解耦里风险最低的第一步。

### PR-B:工作区与历史移到共享存储(为多实例铺路)
- 现在工作区/JSONL 在本地盘。多实例下实例 B 看不到实例 A 的文件。
- 两条路线(本 PR 先**只支持其一并做成可配置**,不强切):
  - **B1 共享卷**(NFS/EFS,改动最小,`CLAUDE_SESSIONS_ROOT` 指向挂载点即可,代码几乎不动)。
  - **B2 对象存储**(S3/MinIO,我们已有 MinIO;改动大,要把读写 JSONL/工作区抽象成存储层)。
- ⚠️ **这步要你定 B1 还是 B2**:B1 快但依赖基础设施提供共享卷;B2 工程量大但更云原生、更贴合
  你的 on-premises + MinIO 现状。**建议 B1 起步**(先解锁多实例),B2 留到真要弹性伸缩时。
- **验证**:两个 ws-server 实例指向同一存储,实例 1 写文件、实例 2 能读到。

### PR-C:运行态进 Redis(会话↔worker 的归属,不再靠进程内引用)⚠️
- 引入"会话运行态"表/键(Redis):`sessionId → {状态:idle/running, 所在实例, worker 句柄}`。
- abort、"是否有正在跑的 worker" 等判断改为查 Redis,而非 `ws.workerProcess`。
- ⚠️ **单向门**:这步改了 abort/生命周期语义,是解耦里最需要你审的设计点。
- **验证**:模拟 abort 落到"非持有连接的实例",确认能正确终止目标 worker。

### PR-D:worker 改为队列驱动(BullMQ)⚠️ 最大一步
- 网关不再直接 `spawn` worker,而是把任务入 BullMQ;worker 池消费、执行、把事件经 Redis pub/sub
  推回网关 → 客户端。
- ⚠️ 这步触及核心数据流,**强制要你的设计 checkpoint**;且它和 ③(worker 用哪种执行后端)耦合
  —— 所以 **PR-D 建议放在 ③ 决策之后**。
- **验证**:网关与 worker 池分进程部署,完整跑通 注册→对话→工具→文件,并发若干会话不串话。

### PR-E:网关无状态化收尾 + 多实例冒烟
- 去掉所有剩余进程内会话假设;起 2 个网关实例 + 1 个共享栈,做多实例冒烟。
- **验证**:轮询/随机把请求打到不同网关实例,同一会话仍正确。

## 4. 分期与依赖

- **PR-A、PR-B(B1)** 与 ③ **完全无关**,可立即做,纯收益(解锁多实例的前两块拦路石)。
- **PR-C** 与 ③ 弱相关,可做,但 abort 语义需你审。
