# Sandbox 隔离实施路线图

**制定时间**: 2025-01-11
**当前状态**: 基础工具可用性已完成（bash + ripgrep），安全边界待实施

---

## 🎯 总体目标

构建**多层安全边界**，实现：
1. **多租户隔离**：用户 A 不能访问用户 B 的数据
2. **会话隔离**：控制跨会话访问行为
3. **系统保护**：防止访问敏感系统路径
4. **审计能力**：记录所有访问行为

### 任务 1.1: 实现跨用户隔离

**目标**：用户 A 不能访问用户 B 的任何数据

**威胁模型**：
```
用户 A 的工作目录：
/data/users/W4dBHOx5UNQdSi0m4md1eWbkIHf83ZFb/sessions/...

用户 B 的工作目录：
/data/users/AnotherUserId/sessions/...
```

攻击者尝试：
```javascript
Read("/data/users/AnotherUserId/sessions/.../file")  // ❌ 必须阻止
Bash("cat /data/users/AnotherUserId/...")            // ❌ 必须阻止
```

**实施方案**（基于 permissionMode: 'default'）：

```javascript
const queryOptions = {
  permissionMode: 'default',
  allow: {
    directories: [
      workspace,                           // 当前会话 workspace
      `${userRoot}/sessions/`,             // 当前用户的所有会话
      '/app/',                             // 项目源码
      `${claudeHome}/.claude/`,            // 用户技能
    ],
  },
  deny: {
    directories: [
      // 关键：阻止其他用户目录
      '/data/users/',
    ],
  },
  // 特殊处理：允许当前用户，阻止其他用户
  // （通过 allow/deny 的顺序实现）
};
```

**关键逻辑**：
1. allow 规则先匹配（允许 `/data/users/{currentUser}/`）
2. deny 规则后匹配（阻止 `/data/users/` 其他用户）

**验收标准**：
- [ ] 用户 A 无法 Read 用户 B 的文件
- [ ] 用户 A 无法 Bash 访问用户 B 的目录
- [ ] 用户 A 可以访问自己的所有会话
- [ ] 日志记录所有跨用户访问尝试（被阻止的）

**依赖**：

- 任务 0.2（选择方案）

### 任务 2.1: 路径遍历防护

**问题**：简单的 `startsWith` 检查可以被绕过

**攻击示例**：
```javascript
// 当前 workspace: /data/users/user/sessions/123/workspace

Read("../../../etc/passwd")
// 解析后: /etc/passwd ❌ 绕过白名单

Read("/data/users/user/sessions/123/workspace/../../../etc/passwd")
// 解析后: /etc/passwd ❌ 绕过白名单
```

**实施方案**：

```javascript
import { resolve, normalize } from 'path';

const isPathSafe = (inputPath, allowedBase) => {
  // 1. 标准化路径（解析 .. 和符号链接）
  const normalized = resolve(allowedBase, inputPath);

  // 2. 严格的前缀检查（必须以 allowedBase + '/' 开头）
  if (!normalized.startsWith(allowedBase + '/')) {
    return {
      safe: false,
      reason: `Path ${inputPath} (resolved to ${normalized}) is outside allowed base ${allowedBase}`,
    };
  }

  return { safe: true, normalized };
};

// 使用示例
const check = isPathSafe("../../../etc/passwd", workspace);
if (!check.safe) {
  console.error(`[Security] ${check.reason}`);
  return false;
}
```

**验收标准**：
- [ ] `../../../etc/passwd` 被阻止
- [ ] `/data/users/../etc/passwd` 被阻止
- [ ] 合法的相对路径正常工作
- [ ] 单元测试覆盖各种路径遍历场景

**依赖**：任务 1.3

**方案 A：禁用符号链接**（严格模式）

```javascript
import { lstat, readlink } from 'fs/promises';

const checkSymlink = async (path) => {

  if (stats.isSymbolicLink()) {

    if (!isPathSafe(target, allowedBase)) {
      return { safe: false, reason: 'Symlink points outside allowed directory' };
    }
  }

  return { safe: true };
};
```

### 任务 2.3: 前缀碰撞防护

**问题**：用户 ID 可能是其他用户 ID 的前缀

**示例**：
```
用户 A: /data/users/user/
用户 B: /data/users/user123/
```

如果检查逻辑不严格，用户 A 可能访问用户 B 的目录。

**实施方案**：

```javascript
const isUserPath = (path, currentUserId) => {
  const normalized = normalize(path);

  // 提取路径中的用户 ID
  // /data/users/{userId}/sessions/...
  const match = normalized.match(/^\/data\/users\/([^/]+)\//);

  if (!match) {
    return { valid: false, reason: 'Not a user path' };
  }

  const pathUserId = match[1];

  // 严格匹配用户 ID
  if (pathUserId !== currentUserId) {
    return {
      valid: false,
      reason: `Path belongs to user ${pathUserId}, not ${currentUserId}`,
    };
  }

  return { valid: true, userId: pathUserId };
};
```

**验收标准**：
- [ ] user 无法访问 user123 的目录
- [ ] user123 无法访问 user 的目录
- [ ] 单元测试覆盖各种用户 ID 组合

**依赖**：任务 1.1

### 任务 3.1: 启用 sandbox-runtime（OS 级隔离）

**目标**：使用 bubblewrap 提供进程级隔离

**实施方案**：


**关键配置**：
1. **enabled: true** - 启用 bubblewrap 沙盒
2. **autoAllowBashIfSandboxed** - 自动允许 sandboxed bash
3. **network** - 网络配置

**sandbox 能做什么**：
- ✅ 限制进程只能访问特定目录（OS 级强制）
- ✅ 限制网络访问
- ✅ 防止进程逃逸
- ✅ 隔离进程命名空间

**sandbox 不能做什么**：
- ❌ 自动区分不同用户（仍需配置）
- ❌ 自动区分读写权限（仍需 permission rules）

**验收标准**：

### 任务 3.3: hooks.PreToolUse 双重检查

**目标**：即使 bypassPermissions，仍通过 hooks 强制执行安全策略

**实施方案**：

```javascript
hooks: {
  PreToolUse: [{
    hooks: [async (input) => {
      if (input.hook_event_name === 'PreToolUse') {
        const { tool_name, tool_input } = input;

        // 1. 路径白名单检查
        if (['read', 'write', 'edit', 'glob', 'grep'].includes(tool_name)) {
          const path = extractPath(tool_input);

          if (!isPathAllowed(path)) {
            return {
              console_text: `❌ Access denied to ${path}`,
              exit_code: 1,
            };
          }
        }

        // 2. Bash 命令检查
        if (tool_name === 'bash') {
          const command = tool_input.command || '';

          if (containsSuspiciousCommand(command)) {
            return {
              console_text: `❌ Suspicious command detected: ${command}`,
              exit_code: 1,
            };
          }
        }

        // 3. 网络访问检查
        if (['web_search', 'web_fetch'].includes(tool_name)) {
          const url = tool_input.url || tool_input.query || '';

          if (!isDomainAllowed(url)) {
            return {
              console_text: `❌ Domain not allowed: ${extractDomain(url)}`,
              exit_code: 1,
            };
          }
        }
      }

      return { continue: true };
    }],
  }],
},
```

**验收标准**：

### 任务 4.1: 结构化安全日志

**目标**：所有安全事件统一格式，便于分析

```typescript
interface SecurityEvent {

  userId: string;
  sessionId: string;
  tool: string;
  path?: string;
  domain?: string;
  command?: string;
  reason: string;
  severity: 'info' | 'warning' | 'critical';
```
