# 第 13 篇：单机 50 并发 —— semaphore(max 8) + worker heap cap + idle reaper

> "16GB / 8 核单机扛 50 并发会话" 怎么算出来的？关键在一个区分：**并发会话数 ≠ 并发执行数**。50 个会话大多在发呆，真正同时烧 CPU/内存的永远 ≤ 8 个——这道闸门就是一个 FIFO 信号量。再加单 worker 堆硬顶和空闲回收，单机预算就守住了。

**章节跳转：**[问题](#问题陈述) · [朴素方案](#朴素方案为什么不行) · [核心方案](#核心方案三道闸) · [实现要点](#关键实现要点) · [反直觉结论](#反直觉结论) · [生产坑](#三个生产坑)

## 问题陈述

OxyGenie 是自托管、单组织、多用户的工作台——典型部署是一台 16GB / 8 核的 VPS，给几十个同事用。第 03 篇定下了 per-message worker：每条用户消息 spawn 一个全新子进程，`--max-old-space-size=1536` 给它一个 1.5GB 的堆。现在把这两个数字摆一起算账：

几十个用户同时在线，假设某一刻一波 20 条消息几乎同时到达。如果来一条就 spawn 一条，**20 个 worker × 1.5GB = 30GB**——16GB 机器当场 OOM，所有人的会话一起死。

但"那就限并发、排队"又不能矫枉过正：如果排队排到天荒地老，用户发一句话等半分钟才开始有反应，自托管工作台的体验就崩了。

所以问题是个双约束：**怎么保证任何一波并发都不把机器 OOM，同时又不让用户排队排到失去耐心？** 答案不在"换更大的机器"，而在看清一件事——这几十个会话里，真正在烧资源的从来只有一小撮。

## 朴素方案为什么不行

**方案一：不限并发，spawn 了再说。** 最省事——WebSocket 收到消息就 `spawn`。但这等于把内存上限交给运气：平峰没事，一旦赶上一波同时发消息，20 个 worker × 1.5GB = 30GB 直接顶穿 16GB。**机器 OOM 是全局事件**——内核的 OOM killer 不会只杀那一波里最贪的 worker，它可能把 ws-server 主进程也带走，所有在线会话陪葬。把内存安全寄托在"用户不会同时发消息"上，是赌一个迟早会输的局。

**方案二：每会话常驻一个 worker（warm pool）。** 第 03 篇已经驳过：50 个活跃会话 = 50 个常驻进程，哪怕 90% 在发呆，每个进程的 SDK 初始化 + MCP 连接 + 常驻堆也要 150～300MB，光发呆就吃掉十几 GB。这条路把"并发执行的内存"问题，换成了"常驻发呆的内存"问题——账更难看。它是 100+ 真并发才值得的优化，不是 50 并发的起点。

**方案三：对"连接数"限流。** 既然怕并发太多，那就限制同时在线的 WebSocket 连接——比如最多 30 个标签页。但这限错了对象：**绝大多数连接根本不在烧资源**，它们只是开着页面在发呆、占一个几 KB 的 socket。你为了挡住那"同时执行"的尖峰，却把"同时在线"的人数砍了，体验代价巨大、防护效果却差——因为 30 个连接照样能在同一刻都点发送，尖峰一点没削。

三个方案的共同教训：**真正稀缺的资源不是"连接"，是"执行槽"。** 内存是被正在跑 `query()` 的 worker 吃掉的，不是被发呆的连接吃掉的。不限并发=赌运气；常驻池=为发呆付费；限连接数=限错了维度。正确的做法是找到那个真正稀缺的东西——同时在跑的 worker 数——只对它设闸。

## 核心方案：三道闸

OxyGenie 的并发预算靠三道闸守住，核心是第一道：

> **闸一是一个容量 8 的 FIFO 信号量，焊死"同时在跑的 worker 数"≤ 8；闸二给单个 worker 一个 1.5GB 的堆硬顶，让失控的那一个吃不垮整机；闸三是空闲连接回收，给长尾僵尸标签页一个出口。** 三道闸合起来，把"50 会话在线"和"≤8 执行、≤12GB"这两件事彻底解耦。

```
50 会话在线
  └─ 大多发呆（占 WS 连接，不占 worker）
  └─ 真正执行 ≤ 8（semaphore 名额）
        └─ 每个 ≤ 1.5GB 堆（heap cap）
              └─ 8 × 1.5 ≈ 12GB < 16GB ✅
```

逐道看它挡住了什么：

- **闸一 · FIFO 信号量（`MAX_CONCURRENT_WORKERS=8`）。** 这是总闸。spawn 之前先 `acquire()`：有名额就拿、立刻 spawn；没名额就在信号量内部的 FIFO 队列里 `await` 等名额。worker 的 `'close'` 事件触发 `release()`，把名额还回去，唤醒队首的等待者（第 03 篇详述了它的生命周期与"名额必须在 close 里还"的铁律）。关键认知：**它限的是"同时在跑的 worker 数"，不是连接数、也不是外部任务队列**。第 9 条消息不是被拒绝，是在队列里 FIFO 等名额——既不 OOM，也不饿死。这一道闸把上面那个 30GB 的灾难场景直接掐死在 8 × 1.5GB。

- **闸二 · 单 worker 堆硬顶（`WORKER_MAX_OLD_SPACE_MB=1536`）。** 信号量管"几个在跑"，堆顶管"每个最多吃多少"。spawn 时给 worker 传 `--max-old-space-size=1536`，单个失控 worker（用户让模型吐 50MB HTML、grep 命中十万行）最多吃到 1.5GB 就被 V8 拦下、自己 OOM 退出——而它的 OOM 是**局部事件**，只杀自己这一个子进程，主进程和别人的会话毫发无损。两道闸一乘：**8 × 1.5GB ≈ 12GB**，给 16GB 机器留出 4GB 余量给 ws-server、Postgres、系统。这个 12GB < 16GB 就是"50 并发能在单机成立"的全部算术。

- **闸三 · idle reaper（空闲回收）。** `shouldReapIdle()` 判定 `(now - lastActivityAt > idleTimeoutMs) && !hasActiveWorker` → 回收空闲连接。注意它的两个条件都要满足：既久未活动、又没有正在跑的 worker，才回收，绝不会误杀正在执行的会话。默认 `idleTimeoutMs<=0` 即**关闭**——小用户量下没必要开，几十个发呆连接的开销可以忽略；它是为更大用户量准备的长尾清道夫。

三道闸的分工很干净：信号量管并发数、堆顶管单体内存、reaper 管长尾连接。而它们守的预算（12GB < 16GB），本质是把第 03 篇那个无状态执行单元的红利兑现成了容量规划——**因为 worker 无状态、用完即弃，你只需要给"同时在跑的"那一小撮设闸，发呆的那一大片几乎免费。**

## 关键实现要点

| 文件 | 行号 | 机制 |
|------|------|------|
| `ws-server.mjs` | L45–58 | `MAX_CONCURRENT_WORKERS` 默认 8 |
| `ws-server.mjs` | L61–72 | `WORKER_MAX_OLD_SPACE_MB` 默认 1536 |
| `src/server/concurrency/semaphore.js` | L11–60 | FIFO 信号量 acquire/release |
| `src/server/concurrency/idle-reaper.js` | L30–35 | `shouldReapIdle()`（默认关） |

这张表里值得点破的是：**这三道闸全是"配置即上限"，没有一行是动态弹性逻辑。** `MAX_CONCURRENT_WORKERS` 和 `WORKER_MAX_OLD_SPACE_MB` 是两个写死的数字，信号量是一段几十行的 FIFO 实现，reaper 是一个布尔判定函数。没有自动扩缩容、没有内存压力探测、没有动态调参——因为对"单机自托管"这个目标来说，可预测、可推演的固定上限，比聪明但难以推理的弹性策略更值钱。你能在部署前用一行乘法（8 × 1.5 = 12）就算清最坏情况，这本身就是稳定性。

`Semaphore(max)` 的 FIFO 语义是这套设计能成立的隐形前提：`acquire()` 返回一个 Promise，没名额就进 `_waiters` 队列；`release()` 时 `shift()` 出最早的等待者唤醒。先到先服务保证了第 9 条消息一定排在第 10 条前面拿到名额，**不会饿死**——这是"排队但体验可接受"的底气。

## 反直觉结论

> [!IMPORTANT]
> **"50 并发"靠的不是更大的机器，而是一个区分：并发会话数 ≠ 并发执行数。**
>
> 用户体验上是"50 个人同时在线"，资源上是"最多 8 个在烧"。一个 8 容量的 FIFO 信号量 + 一个 1.5GB 的堆顶，就把"在线规模"和"执行规模"解耦了。**容量规划的第一步永远是：找到真正稀缺的资源（这里是执行槽），只对它限流，而不是对"连接数"限流。** 限错了对象——比如去砍在线人数——你既伤了体验，又没削掉真正的尖峰，因为尖峰从来不在连接里，在"同时跑 query()"里。看清谁是瓶颈，比堆硬件便宜得多：12GB < 16GB 的安全，是用一行乘法买来的，不是用一台更贵的机器买来的。

## 三个生产坑

> [!WARNING]
> **坑一 —— 这是 per-message spawn，不是 worker 池，~50 真并发就是天花板。**
> 三道闸守的是"单机 16GB 下 50 并发"这个具体预算，它建立在"大多数会话在发呆、真执行 ≤ 8"的假设上。一旦真并发逼近 50——也就是发呆假设不再成立、信号量队列常年排满——用户就会明显感到排队延迟，而 per-message 模型每次都要付冷启动成本，没有常驻池摊薄。这时候需要的就是第 03 篇里"100+ 真并发才值得"的 warm pool（未来）。当下的 50 是设计上限，不是可以无脑往上加的数字；要扩，得换执行模型，不是调大信号量。

> [!WARNING]
> **坑二 —— 没有硬磁盘配额，写满磁盘没有硬拦。**
> 三道闸守住了内存和执行并发，但**磁盘是没设闸的那一维**。Bash runner 是跑后测量、只告警（第 10 篇），没有 `prlimit` 式的硬配额在前面挡。一个会话让模型疯狂写文件、或预览反复 `npm install` 堆 node_modules，理论上能把宿主磁盘写满——而磁盘满是另一种全局故障，会牵连所有会话和 Postgres。内存有 1.5GB 硬顶，磁盘目前只有"事后发现"。补救方向是给 workspace 加硬配额，在此之前小用户量靠监控盯着。

> [!WARNING]
> **坑三 —— idle reaper 默认不开，僵尸标签页不会被自动回收。**
> `idleTimeoutMs<=0` 时 `shouldReapIdle()` 恒返回 false，闸三实际处于关闭状态。后果：用户开着页面走了、前端还在自动 ping 维持连接的"僵尸标签页"，会一直占着 WebSocket 连接不被清理。小用户量下这完全可接受——几十个发呆连接不值得为它引入回收逻辑的复杂度；但用户量上来、僵尸连接累积时，得显式把 `idleTimeoutMs` 设成正值开启回收。这是个刻意留的"默认关、按需开"开关，不是 bug。

三个坑的共同根源是：**三道闸是为一个具体预算（单机 16GB、~50 并发、内存维度）精确调出来的，它的稳，恰恰来自它不试图覆盖所有情况。** 超出并发上限要换执行模型（坑一）、磁盘维度还没设闸（坑二）、长尾回收默认不开（坑三）——每一个都是"为可部署性优先于超大规模"这个北极星主动做的取舍。知道闸守住了什么、没守住什么，才能在用户量增长时踩准下一步该补哪道闸。
