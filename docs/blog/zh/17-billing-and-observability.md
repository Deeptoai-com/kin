# 第 17 篇：计费与可观测 —— usage_record 观测、为什么 costUsd 不扣费、PostHog/Sentry/审计三件套

> OxyGenie 跑在字节 Code plan 的**包月 ¥200 固定**上，所以 SDK 报的 `costUsd`（按 Anthropic 公开价估的）**和真实花费毫无关系**。这一篇讲一个"还没开始收钱"的计费系统怎么设计：先观测落库、攒数据、定换算率，再开配额；以及可观测三件套（PostHog 看行为、Sentry 看错误、audit_log 看安全）。

## 问题陈述

OxyGenie 是自托管、单组织、多用户的工作台。"多用户"立刻带出一个现实约束：一个 ¥200/月 的 ARK 包月额度，是**全组织共享**的。一个用户开着 agent 通宵跑大任务，把这个月的额度烧光，其他同事就用不了了。所以"公平配额"是迟早要做的——但要做对，得先回答两个互相缠绕的问题：

1. **拿什么计量？** SDK 在每个 `result` 事件里报一个 `costUsd`，看起来是天然的计费基准。但 OxyGenie 付给上游的是**包月 ¥200 固定**，不是按调用计费。`costUsd` 是 SDK 按 **Anthropic 公开价**估出来的——我们既不走 Anthropic、也不按量付费，这个数字和真实花费**毫无关系**。拿它扣费等于拿一把刻度错的尺子量布。
2. **什么时候开始拦？** 配额 gate 一旦上线就会拦人。但拦的阈值从哪来？在你不知道"一次典型 run 到底烧多少"之前定的任何阈值，要么太松（拦不住失控用户）、要么太严（误伤正常使用）。

把这两个问题摆一起，"做计费"就不再是"接个支付、写个扣费逻辑"那么简单。它先是一个**度量问题**：在一个计价模型和上游错配、真实成本不可见的环境里，你要先建立起对"用量"的可信观测，才谈得上公平地分配和收费。再叠上一层：多用户系统出了安全事件（谁删了什么、谁越权了），也得能查得到、查得准、查得到底——这是可观测的第三条腿。

## 朴素方案为什么不行

**方案一：直接按 SDK 的 `costUsd` 扣费。** 最省事——SDK 都替你算好了，落库、累加、超额拦截。但前面说了，`costUsd` 是按 Anthropic 公开价估的，而我们付的是 ARK 包月。两个数字活在不同的计价宇宙里：可能 SDK 报"这次花了 $0.8"，而我们这个月无论跑多少都是 ¥200。拿它扣费，用户的账单和你的真实成本对不上，配额阈值也建在流沙上。**计价基准和上游付费模型错配，是这个方案的死穴。**

**方案二：先不管校准，立刻上配额。** 既然迟早要拦，不如现在就拍一个阈值（比如"每人每天 100 次 run"）先拦着。问题是这个阈值是**猜**的：拍高了等于没拦，拍低了正常用户写两个功能就被卡住，体验崩。**在没有真实用量分布之前上配额，本质是拿用户做计费实验**——你用线上用户的挫败感，去试探一个本该用数据算出来的数字。

**方案三：审计表外键到 user，跟着用户表走。** 建 `audit_log` 时，很自然会把 `userId` 设成指向用户表的外键——数据库范式、引用完整性、级联删除，教科书都这么教。但这恰恰毁掉了审计的核心价值：**当一个用户被删除时**（离职、清理、甚至就是他干了坏事之后跑路），外键约束会让审计记录跟着孤儿化或被级联删掉。而取证最需要的就是"这个已经不存在的用户当时做了什么"。**把审计绑死在用户表的生命周期上，等于在最需要它的时候把它销毁。**

三个方案的共同教训：**计费和审计的难点都不在"写逻辑"，在"基于什么、在什么时候、以什么独立性去记录"。** 按 `costUsd` 扣费错在基准；立刻上配额错在时机；审计外键错在把可变的引用完整性看得比不可变的取证价值更重。OxyGenie 的答案是把这三件事各退一步：计费先只观测不收钱、计量用 token 不用 costUsd、审计故意脱钩用户表。

## 核心方案：观测优先 + token 为基 + 三件套

OxyGenie 当前的策略可以一句话概括：

> **计费系统先建"观测"再建"扣费"：`usage_record` 当只读账本先跑，以可信的 token 为计价基（costUsd 仅参考），攒够真实用量再反推换算率、再开配额。可观测分三件套——PostHog 看行为、Sentry 看错误、append-only 的 audit_log 看安全（故意不外键到 user）。**

逐段看它怎么把上面三个坑各个击破：

**① 观测优先（P2-1，当前阶段）—— `usage_record` 是只读账本，不是扣费器。** 每个模型、每个 run 落一行：`inputTokens` / `outputTokens` / `costUsd`（仅参考）/ `numTurns` / `isError`。它现在**只观测、不收钱**。这一步的目的是攒数据：跑上 1–2 周，就能从真实分布里反推出"¥200 ≈ X token ≈ Y run"，得到一个**有数据支撑的换算率**，然后才去开配额。这就是对方案二的回答——配额的阈值不是拍出来的，是从这张只读账本里算出来的。

**② token 为计价基，credit 为抽象单位 —— 用可信的那个量。** 计量基准选 token，不选 costUsd。理由很简单：**token 数来自上游 API，是上游真实返回的事实，可信**；costUsd 是 SDK 二次估算的，在我们的部署形态下不可信。再往上抽一层：对用户暴露的是 `credit`（抽象单位），`1 credit = N token` 可配、可按模型设不同换算率——这样将来换模型、换网关，对用户的计价口径不用动，只改换算率。costUsd 始终只做"参考显示"，永远不进扣费链路。

**③ 计费表已就位 —— 机制先建好，开关后打开。** 库表是齐的：`plans`（套餐）/ `subscriptions`（订阅）/ `credit_balances`（`monthlyAllotment` 月度配额 + `extraCredits` 加购）/ `credit_ledger`（流水，记 `delta` / `kind` / `meta`）/ `invoices`（账单）；订阅状态由 Polar webhook 接入。注意这是"机制就位、未接生产流"的状态——表能记账，但还没有 gate 真去拦（见坑二）。先把数据模型立住，等换算率定了，开配额就是接上一段 gate 逻辑的事，不用重构。

**④ 可观测三件套 —— 行为、错误、安全各看一眼。** 三条独立的观测线，各管一摊：
- **PostHog（行为）**：约 7 类行为事件，`sessionId` 截前 8 位脱敏后上报——看用户怎么用、在哪一步流失，不碰敏感内容。
- **Sentry（错误）**：JS 前端和后端异常都进 Sentry，看"哪里崩了"。
- **`audit_log`（安全）**：append-only 的安全审计，记 `action` / `target` / `meta` / `ip`，`userId` **不是外键**。它要在用户被删之后仍然活着——这是对方案三的直接回答：取证价值要求审计独立于用户表的生命周期。

三件套是刻意分开的：行为分析、错误监控、安全取证是三种不同的关切，混在一张表里会互相拖累（行为事件高频、错误要去重、审计要不可变）。分开各用合适的工具，是这套观测的设计骨架。

## 关键实现要点

| 文件 | 行号 | 机制 |
|------|------|------|
| `src/db/schema/usage-record.schema.ts` | L24–67 | 每模型一行；`costUsd numeric(14,6)`，**注释明示不用于扣费** |
| `src/server/usage/build-usage-rows.ts` | L49–83 | SDK `result` → 每模型一行（无则 'unknown' 兜底） |
| `src/db/schema/billing.schema.ts` | L52–101 | credit_balances / credit_ledger / invoices |
| `src/lib/observability/posthog-events.ts` | L7–99 | 行为事件 + `sessionId.slice(0,8)` 脱敏 |
| `src/db/schema/audit-log.schema.ts` | L17–46 | append-only，userId 非 FK，action/target/meta/ip |

观测落库的入口在 `build-usage-rows.ts`（L49–83）：它把 SDK 一个 `result` 事件**拆成每模型一行**——一次 run 如果主模型 + 子 Agent 用了不同模型，就落多行，模型字段缺失时用 `'unknown'` 兜底，绝不丢记录。落库目标是 `usage-record.schema.ts`（L24–67），这里有一个值得专门指出的细节：`costUsd` 的列类型是 `numeric(14,6)`，而 **schema 注释里明确写了"不用于扣费"**——把"这个字段不可信"这件事钉死在数据模型层，而不是只留在某个工程师的脑子里，是为了防止将来有人顺手拿它去做扣费 UI（这正是坑一）。

计费表在 `billing.schema.ts`（L52–101）：`credit_balances` 区分 `monthlyAllotment`（月度配额，每月刷新）和 `extraCredits`（加购，不随月清零）；`credit_ledger` 是流水账，每笔记 `delta`（增减）/ `kind`（类型）/ `meta`（上下文）。可观测侧，`posthog-events.ts`（L7–99）集中定义行为事件并统一做 `sessionId.slice(0,8)` 脱敏；`audit-log.schema.ts`（L17–46）是那张 append-only 的审计表——`userId` 是普通列不是外键，配 `action` / `target` / `meta` / `ip`，专为"用户删了也要留痕"设计。

## 反直觉结论

> [!IMPORTANT]
> **计费系统最该先建的不是"扣费"，是"观测"。**
>
> OxyGenie 把 `usage_record` 当一本"只读账本"先跑起来——token 可信、costUsd 不可信，先把真实用量攒够，再反推换算率、再谈配额。**在你知道"一次典型 run 到底烧多少"之前就上配额，是拿用户做实验。** 大多数人做计费的本能顺序是"接支付 → 写扣费 → 加配额"，但在一个计价模型和上游错配、真实成本天然不可见的环境里，正确的顺序是反过来的：先观测、再校准、最后才动钱。扣费逻辑是几十行的事，难的是那把"换算率"的尺子——而尺子只能从真实数据里量出来，量不出来就先别拦。

同一种"把记录的独立性看得比方便更重"的判断，也体现在审计表上：**`audit_log` 故意不外键到 user。** 范式会告诉你该加外键、该级联删除——但审计的全部价值，在于"那个已经不存在的用户当时做了什么"。如果审计跟着用户一起被删，它就在最需要的时刻消失了。所以这里的取舍是明确的：**合规与取证场景里，不可变性（immutability）> 引用完整性（referential integrity）。** 审计是历史，历史不该因为现在的实体被删除就被改写。这和计费"先观测"是同一种工程品味的两面——**先保证你记下的东西是可信的、是会活下来的，再谈拿它去做决策。**

## 三个生产坑

> [!WARNING]
> **坑一 —— `costUsd` 不是真实花费，任何计费 UI 都要标清。**
> `costUsd` 是 SDK 按 Anthropic 公开价估的，而 OxyGenie 付的是包月 ARK——这个字段和我们的真实成本不在一个计价宇宙里。schema 注释已经写明"不用于扣费"，但注释挡不住有人在做后台时顺手把它渲染成"本月花费 $X.XX"给用户看。一旦这么做，用户会拿这个假数字质疑你的账单、你也会拿它做错误的成本决策。**铁律：costUsd 只能出现在标了"仅供参考、非真实账单"的地方，扣费链路永远用 token/credit。** 在 schema 层钉死注释只是第一道防线，UI 层得有第二道。

> [!WARNING]
> **坑二 —— 当前无配额强制，能记账但没人拦。**
> 计费表全就位：`credit_ledger` 能记每一笔增减，`ensureDailyRefill` 这类配额维护逻辑也写好了。但它们**尚未接入生产流**——也就是说，现在系统能**记录**用量、能算余额，却没有一个 gate 在 run 开始前去**检查并拦截**超支。后果是：一个用户真要烧光 ¥200 共享额度，目前没有任何东西拦得住他。这是"观测优先"路线的代价，也是有意为之的阶段状态——配额引擎要等换算率校准完才接上。但别误以为"计费表都建好了 = 已经在收费/限额了"，当前是只记不拦。

> [!WARNING]
> **坑三 —— audit_log 无保留策略会无限增长，costUsd 经 Drizzle 是字符串要小心聚合。**
> 两个长期隐患。其一，`audit_log` 是 append-only 的，**没有保留/清理策略**——它只增不删，长期会一直长大；为取证选的"永不删除"，反过来就是"无限增长"，迟早要补一个分区归档或冷存储策略。其二，`costUsd` 是 `numeric` 列，经 Drizzle 读出来是**字符串**而不是数字。任何对它做 `SUM`/`AVG` 的聚合（哪怕只是"仅供参考"的统计图），都要先显式转类型——直接当数字加会得到字符串拼接或 NaN，是个安静的类型陷阱。这两个坑都不致命，但都属于"上线时不报错、跑久了才咬人"的那一类，提前记下。

这三个坑的共同根源是：**OxyGenie 的计费当前处在"度量已建、决策未接"的中间态。** 表建好了、数据在落、观测在跑，但扣费、配额、清理这些"拿数据去做动作"的环节都还空着。这不是没做完，是按"先观测后动作"的路线有意停在这一步。把"能记"和"已拦"分清楚，把"参考值"和"真账单"分清楚，是用这套系统时最该绷紧的两根弦。
