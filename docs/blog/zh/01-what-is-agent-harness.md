# 第 01 篇：什么是 Agent Harness —— 官方 SDK 已经给了 Loop，你还要造的那 15 层

> 蓝本 HarWork 第 01 篇问"什么是 Harness"，并自己写了 20 行 Loop 骨架。OxyGenie 的答案要先撇清一件事：**Loop 不用你写——Claude Agent SDK 的 `query()` 已经把"调 LLM → tool_use → 执行 → 回灌 → 再调"这套循环、上下文压缩、token 记账全做好了。** 那"造 Harness"到底在造什么？答案是：把这个**无状态、单机、单用户**的 SDK 调用，变成**多租户、可控、可观测、可计费、可上线**的运行时层——大约 15 层。本文回答：当内核已经免费，外面那 15 层为什么一层都省不掉？

## 问题陈述

`query({ prompt, options })` 三行就能跑一轮会带工具的对话：模型自己决定调哪个工具、SDK 自己解析 `tool_use`、自己执行、自己把结果回灌、自己接着再调，直到吐出 `result`。这是 Anthropic 替你写好的 Agent Loop，它甚至帮你做了上下文压缩和 token 记账。那为什么 OxyGenie 还有三万多行 TypeScript？

因为"能跑一轮 demo"和"能给一个团队当生产工具"之间，隔着整整一圈工程。demo 里那个 `query()` 默认是**无状态**的（这一轮跟上一轮没关系）、**单机**的（就在你这个 Node 进程里跑）、**单用户**的（没有"别人"这个概念）。而 OxyGenie 的定位是**自托管、单组织、多用户**的 Claude-Agent 工作台：一屋子同事同时开着浏览器，每个人都可能随手发一条会跑 Python、写文件、起服务、可能跑五分钟的消息。把"无状态单机单用户"喂给"有状态多机多用户"的现实，每一个差距都是一层必须补的工程。Agent Harness，就是补这一圈差距的那层壳。

## 朴素方案为什么不行

最朴素的做法是：把 `query()` 包一个聊天前端，连上 WebSocket，直接上线。它不会立刻报错——恰恰相反，第一次 demo 会跑得很漂亮。问题是它会**依次**死在下面每一个地方，而且每一个都不是"上线前修一下"能补的，全是架构层面就得提前规划的洞：

- **丢上下文**——`query()` 这一轮的对话不会自动落盘。用户刷新浏览器，历史就归零；想"接着上次聊"，SDK 的 transcript 根本没人替你保存和索引（→ 第 12 篇）。
- **无隔离**——`query()` 默认跑在你的主进程里，跟它执行的工具共享同一个 V8 堆。一个用户的失控 Python（`while True`、吃满内存、段错误）会把整个 Node 进程带走，**连带所有人的 WebSocket 一起断**（→ 第 03 篇）。
- **并发即崩**——没有任何上限。十几个人同时点发送，十几个 `query()` 一起在堆里跑工具、攒 buffer，一波并发就能把 16GB 机器 OOM 掉（→ 第 13 篇）。
- **断网即丢**——WebSocket 一断，没有续传机制，正在跑的那一轮结果永远拿不回来；用户只能重发，但 token 已经烧掉了（→ 第 04 篇）。
- **删库无人拦**——LLM 是会真的去跑 `rm -rf`、真的去读别的用户工作区文件的。裸 `query()` 没有沙箱、没有 path guard、没有密钥剥离，等于把服务器交给模型（→ 第 10/11 篇）。
- **扩展靠改源码**——想给团队加一个内部工具、一套 Skill、一个 MCP？裸方案里这些全得动核心代码，每次扩展都是一次重构（→ 第 06/07/08 篇）。
- **不能收费、看不见、上不了线**——没有用量记录就没法配额和计费，没有埋点和审计就出了问题查不到，没有部署管线就根本上不了线（→ 第 17/18 篇）。

这七条不是 bug 清单，是**七类系统性缺失**。它们的共同点是：单独看每一条都"以后再说"，合起来却决定了这个东西到底是个玩具还是个产品。把它们补齐，就是把裸 `query()` 升级成 Harness 的全部内容。

## 核心方案：OxyGenie 的 15 层（SDK 在第 0 层）

OxyGenie 的整个架构可以画成一座塔。塔基（第 0 层）是 Claude Agent SDK 的 `query()`——Loop、上下文、工具协议都在这里，**你不用写**。往上 15 层，每一层都对应朴素方案里那个会死的地方：

```
15. CI/CD + 部署（多阶段 Docker / GHCR / Dokploy / Traefik）   → 第 18 篇
14. 可观测 + 审计（PostHog / Sentry / audit_log）              → 第 17 篇
13. 计费与配额（usage_record 观测 / token-based credits）      → 第 17 篇
12. 真预览（per-session Docker + 子域反代 + JWT）              → 第 15 篇
11. 会话 UI / Workbench（turn card / artifact / seq 排序）     → 第 16 篇
10. 多模型路由（ARK 网关 / 模型别名 / 钉版）                   → 第 14 篇
 9. 并发与回收（semaphore / heap cap / idle reaper）           → 第 13 篇
 8. 会话持久化（SDK transcript 为真相 + DB 13 表索引）         → 第 12 篇
 7. 多租户隔离（per-session workspace + path guard）           → 第 11 篇
 6. Bash 沙箱（srt/Docker + prlimit + secret strip）          → 第 10 篇
 5. 权限与 HITL（Ask/Act + canUseTool 审批）                  → 第 09 篇
 4. Skills / MCP / 工具（preset + 自定义 MCP + FS 启用）       → 第 06-08 篇
 3. ExecutionRuntime 抽象（local / docker 双后端）             → 第 05 篇
 2. 流式协议（NDJSON + seq + 背压）                            → 第 04 篇
 1. Per-Message Worker（每条消息 spawn 子进程跑 query()）      → 第 03 篇
─────────────────────────────────────────────────────────────
 0. Claude Agent SDK 的 query()（Loop / 上下文 / 工具协议）    ← 不用你写
```

**核心定义**：Agent Harness = 把第 0 层那个"无状态、单机、单用户"的 SDK 调用，用 1～15 层包成一个"多租户、可控、可观测、可计费、可上线"的产品运行时。

读这张塔的关键是看清两件事。第一，**层与层之间不是平铺的功能清单，而是有依赖的承重结构**。最底下三层（Per-Message Worker、流式协议、ExecutionRuntime 抽象）解决"`query()` 该在哪儿跑、怎么把它吐的字流回浏览器、本地还是容器跑"——这是地基，上面所有层都站在它身上。中间几层（工具/权限/沙箱/隔离/持久化）解决"跑的时候安不安全、跑完记不记得住"。最上面几层（并发回收、多模型、会话 UI、真预览、计费、可观测、部署）解决"能不能撑住一屋子人、能不能上线、能不能被采购"。抽掉任何一层，它上面那些层都会塌——你没有第 1 层的进程隔离，第 13 层的并发回收就无从限起；你没有第 8 层的持久化，第 11 层会话 UI 上的"历史会话"就是空的。

第二，**每一层都不是 over-engineering，它们各自被"用户的某种具体期待"逼出来**。用户期待刷新页面历史还在——逼出第 8 层；期待自己的代码不被别人看到——逼出第 7 层；期待手滑不会被模型 `rm -rf`——逼出第 5、6 层；期待团队能加自己的工具——逼出第 4 层；老板期待这东西能算成本、能进采购流程——逼出第 13、14 层。**没有一层是工程师自嗨加上去的；每一层背后都站着一个会投诉的人。** 这正是 Harness 和"玩具 demo"的本质区别：demo 服务的是"我自己试一下"，Harness 服务的是"一群人指望它干活"。

而第 0 层那个被官方做掉的 Loop，恰恰是这座塔里**最不需要你操心**的一块。这也是整个系列反复要砸实的论点：OxyGenie 没有、也不打算自己写 Agent Loop。它把全部工程预算花在 SDK 之外那一圈——后面每一篇，拆的都是这 15 层里的某一层，没有一篇在教你"怎么写 Loop"。

## 反直觉结论

> [!IMPORTANT]
> **当官方 SDK 把 Loop 做好之后，"造 Agent 产品"的难度并没有下降，只是从"内核"挪到了"外圈"。**
>
> 你省下的是第 0 层那几百行；你仍然要自己写 1～15 层那三万行。**真正决定产品体验的，从来不是"调 LLM 那一下"，而是"让那一下在多租户、断网、并发、越权、要收费的现实里不崩"的全部工程。**

这个结论之所以反直觉，是因为大多数人第一次跑通 `query()` 的瞬间会产生一种错觉：核心都做完了，剩下的不就是套个壳吗？事实正相反——核心被官方做完，意味着**你和所有竞争对手的起点完全一样**，谁也没有"我家 Loop 更聪明"的护城河。差异化只能、且全部发生在外圈：谁的隔离更干净、谁的断网续传更稳、谁的并发上限算得更准、谁的真预览能真跑起来。Loop 是免费的，**外圈才是产品**。

换个角度说：官方 SDK 把"难在算法"的部分免费了，剩下的是"难在工程"的部分——而工程的难，是琐碎、是边界情况、是"50 并发会话怎么在一台 16GB 机器上不崩"这种没有银弹、只能一层一层抠的难。这种难不会因为模型变强而消失，它属于另一个维度。所以这座 15 层的塔，不是历史包袱，而恰恰是这个产品**唯一**能积累壁垒的地方。

## 什么时候才该自建 Harness

把整圈工程摊开之后，一个诚实的问题是：你真的需要自己造这 15 层吗？大多数情况下，答案是**不需要**——直接用别人托管好的 Agent 平台，或者就裸跑 `query()` 做个内部小工具，是更省的选择。自建 Harness 的成本是实打实的三万行加上长期维护，它只在三个条件**同时**成立时才划算：

第一，**你必须自托管**。数据不能出公司、模型必须走自己选的网关（OxyGenie 钉死在 ARK 兼容上）、合规要求代码和工作区都留在自己机器上——这时候"用别人托管的 Agent"这条路直接断了，你只能自己把运行时立起来。

第二，**你是真多用户、真要执行**。如果只是你一个人偶尔问问模型，裸 `query()` 足够，那 15 层全是浪费。但一旦是一屋子同事并发、而且要让模型真的跑代码 / 写文件 / 起服务，"隔离 + 并发 + 沙箱 + 持久化"这四层就从可选变成必须——这正是 OxyGenie 存在的理由。

第三，**它要长成产品，不是脚本**。要能计费、能观测、能审计、能一键部署、能被采购——最上面那几层（13/14/15）只有当这东西要"被一个组织正经用起来"时才有意义。如果它永远是个跑在你笔记本上的脚本，这几层一行都不用写。

反过来看 OxyGenie 的处境：它三条全中——自托管、组织内多用户且重执行、要做成可部署的工作台。所以它没得选，必须把 15 层补齐。但请注意威胁模型的边界：OxyGenie 服务的是**半可信的同事**，不是公网匿名攻击者。这条前提让很多层可以做成"纵深防御 + 警示护栏"，而不是"反匿名攻击的锁死"——触达服务器的强力能力（执行代码、连内网工具）是**合法核心用途**，用沙箱兜底而非一禁了之。**先认清自己是不是这三条都中，再决定要不要造这座塔**——这是读完本系列之前，最该先想清楚的一件事。
