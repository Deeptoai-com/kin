# 第 19 篇：复盘 —— 从 AI SDK starter 到自托管 Agent 平台的得、失与反悔

> 蓝本 HarWork 第 18 篇复盘"49 天独立造 Harness"。OxyGenie 走的是另一条起点：**从官方 AI SDK starter（`claude-agent-chat`）出发，包住 Claude Agent SDK，长成一个多租户、自托管、可计费、有真预览的平台。** 这一篇不讲新功能，讲这条路上**哪些决策回头看是对的、哪些是债、哪些已经在反悔**。

## 一、这套系列覆盖了什么（一句话回看 18 篇）

OxyGenie 的核心命题只有一句：**不自己写 Agent Loop，把工程全花在 SDK 之外的 15 层。** 这句话听起来像偷懒，其实是一个杠杆选择——把"调 LLM → 解析 tool_use → 执行工具 → 回灌 → 再调"这套循环、上下文管理、token 记账，全交给官方 SDK 的 `query()`，省下来的精力不是去休息，而是去填 SDK 够不到的那一圈：执行（03-05）、工具扩展（06-08）、沙箱权限（09-11）、会话并发（12-14）、产物预览（15-16）、计费上线（17-18）。

回头看这 15 层，会发现一个规律：**没有一层是炫技，每一层都是被"多租户产品"的某个具体现实逼出来的。** per-message worker 是被"一个用户的失控代码不能拖垮别人"逼出来的；ExecutionRuntime 的 FAIL-CLOSED 是被"沙箱没就位绝不能裸跑"逼出来的；usage_record 的只观测是被"costUsd 不可信、不能拿用户做计费实验"逼出来的；Origin CA 是被"CF Full(Strict) 不让用 HTTP-01"逼出来的。**SDK 解决了"Agent 怎么想"，这 15 层解决的全是"Agent 怎么在一个真实的、多人的、要上线的、要花钱的系统里活下去"。** 这才是 OxyGenie 真正的工程量所在，也是这套系列想说清的事。

## 二、4 件做对的

**1. 包 SDK，不重写 Loop。** 这是所有决策的根。省下的不只是几百行 Loop 代码——那点代码量其实不大——真正省下的是**"跟随 SDK 演进"的能力**。SDK 每次升级带来的新工具、新上下文管理、新 token 优化，我们自动继承；如果自己写了 Loop，每一次都要追着重新实现一遍。把精力放在 SDK 够不到的那一圈（隔离、持久化、计费、预览、上线），是整个项目里杠杆最高的一笔投入。这一条做对了，后面 15 层才有意义。

**2. per-message 子进程做隔离原语（第 03 篇）。** 用操作系统的进程边界，一次性换来隔离、可中断、状态干净三件事。每条消息 spawn 一个全新子进程、跑完即死——代价只是一次冷启动，回报是"50 并发会话"能在一台 16GB 单机上成立。更深的红利在于：**当执行单元是无状态的，扩容、限流、容错就全部变简单了。** 不用写进程健康检查、不用复位全局变量、不用清半截状态——进程用完即弃，下一条消息是一张白纸。这个"无状态执行单元"的哲学，后来在真预览、在 semaphore 限流里被一次次复用，是整个架构复利的起点。

**3. ExecutionRuntime 抽象 + FAIL-CLOSED（第 05/10 篇）。** 把"在哪执行、怎么隔离"收敛成一个接口，再把"沙箱没就位就拒绝执行"焊进这个接口的契约。FAIL-CLOSED 是一条铁律：宁可拒绝，绝不降级裸跑。它挡住的是分布式系统里最常见、也最致命的一类安全塌方——**"沙箱起不来，那就先不用沙箱跑吧"**。这种"临时降级"在压力下极其诱人，而 FAIL-CLOSED 用一行契约把这条路堵死了。把安全做成"默认拒绝"而不是"默认放行"，是这个项目最值得骄傲的一笔。

**4. 观测优先的计费（第 17 篇）。** 先只读落库、不急着扣费——`usage_record` 当只读账本先跑，攒够真实用量再反推换算率、再开配额。这一条避开了一个很多人会犯的错：**在没校准前就上配额，等于拿线上用户做计费实验。** token 可信、costUsd 不可信，先把可信的量攒够，再谈拿它做决策。这种"先观测、后动作"的克制，和 FAIL-CLOSED 是同一种工程品味——不在没把握的时候对用户出手。

## 三、5 件做错 / 欠的债

**1. 会话 UI 没有 seq（第 16 篇）。** 这是 UI 最大的结构债。事件按到达顺序拼、历史/实时去重靠"内容看起来一样"拍脑袋，`messages_loaded` 还在 adapter 和事件队列里双处理。后果是 resume 后队列偶发卡住、消息重影。根子是一开始没给事件一把统一的排序键——worker 那侧第 04 篇早有 seq，却没一路贯通到前端 store。这债不致命，但它让"会话 UI 已经收口"成了一句不能说的话。

**2. structured outputs 被迫关掉（第 15/16 篇）。** 本想用 SDK 的 `outputFormat` 让模型结构化自报产物，结果它的 Stop-hook 把 `You MUST call StructuredOutput` 漏进了对话上下文——为了一个产物清单，污染了整段对话。只能退回启发式 + manifest 检测，代价是会把解释文字里的代码块误判成产物。这债的特殊之处在于：**它不是我们写错了，是 SDK 的一个特性把内部细节漏了出来。** 它和真预览的 manifest 策略耦合，manifest 没定死之前不解。

**3. per-message worker 与 per-session 预览容器是两套运行时（第 15 篇）。** Agent 执行是"每条消息 spawn、跑完即死"，预览需要"per-session 持久容器、暖 node_modules 缓存"——一个用完即死、一个要常驻暖缓存，两套生命周期、两个管理器，目前并行没合并。后果是会话工作区的文件要在 worker 和预览容器之间共享，边界没理清就会出现"Agent 写了文件、预览没看到"。这是当前最大的架构张力，Phase C 正在设计统一。

**4. Skills copy-on-enable 复制爆炸（第 08 篇）。** Skill 是"启用时拷贝一份到用户工作区"，于是 1 个 Skill × 100 个用户 = 100 份拷贝。小团队还扛得住，但规模一上来就吃磁盘。这是典型的"起步图省事、规模化才咬人"的债——当时选 copy-on-enable 是因为它最直接、最不容易出错，但没给规模留余地。修法是改 DB 目录 + 懒加载，让 Skill 引用而非复制。

**5. resume 的相对路径坑（第 03/12 篇）。** `claudeHomePath` 存了相对路径，叠上 worker 和 ws-server 的双 cwd，resume 时解析出的 transcript 路径对不上，**历史归零**。2026-06-02 才修（DB 里一律存绝对路径），在那之前的旧会话有损失。这是个很小的 bug、很大的教训：**跨进程、跨 cwd 的路径，任何相对都是定时炸弹。** 一个 `.slice` 一样不起眼的相对路径，能让用户的整段历史凭空消失。

## 四、2 个选型反悔（回头看对错参半）

**钉死 SDK 0.2.112 换 ARK 多模型（第 14 篇）。** 这是一笔有明确收益、也有明确代价的交易。收益是：钉在 ARK 兼容上限 0.2.112，换来了 ARK（火山）那套便宜的多模型网关——多模型买到了，成本压下来了。代价是**版本自由**：0.2.113+ 改成了原生二进制，和 ARK 网关不兼容，这套"借壳 SDK"的玩法一旦 SDK 往前走就会断。值不值，取决于一个我们控制不了的变量——**ARK 这条网关还能稳多久**。迁移到原生 Anthropic + native binary 是一件悬在头上、迟早要做的活。回头看，这笔交易没有错，但它把一部分命运交给了上游的版本节奏，这一点要时刻清醒。

**transcript 为真相、DB 为索引（第 12 篇）。** 起步时这是最省的选择：SDK 已经把对话写成 transcript 了，DB 只存个索引指过去，不用自己再维护一份消息真相。但它埋了一个雷——**把别人（SDK）的缓存当成了自己的真相**。transcript 是 SDK 的内部产物，它的路径、格式、存在性都不归我们管；一旦 resume 找不到 transcript（就像上面那个相对路径坑），UI 直接空白，因为我们手里没有第二份真相。回头看，这是个典型的"早期省事、后期还债"的选择，已经规划翻转成"DB 为真相、transcript 为缓存"——让消息的权威副本握在自己手里。

这两个反悔有个共同点：**它们都是"把某个根基交给了上游"的决策。** 一个交了版本，一个交了真相。起步时这么做都对——它们让项目跑得更快、更省。但它们也都是"借来的根基"，借来的东西迟早要还，区别只在于你是否清楚自己借了什么、什么时候得还。

## 五、几个一人/小团队的取舍

做这个项目的人手有限，所以很多决策不是"哪个技术更好"，而是"哪个更省一层复杂度"。三个有代表性的：

**单机 Compose 而非 k8s。** 50 会话的目标下，k8s 的控制平面、调度器、运维心智全是过度工程。一份 Docker Compose 把十来个服务编排清楚，足够了。这不是"看不起 k8s"，是"在这个规模，k8s 的收益填不平它的复杂度"。

**idle reaper 默认关。** 预览容器的空闲回收机制写好了，但默认不开——小用户量下，几个发呆的容器占点内存可以接受，换来的是少一层"什么时候该杀容器"的判断和它可能引入的边界 bug。等用户量真上来了再开，是更划算的顺序。

**计费只观测不收钱。** 不急着做配额引擎，先攒数据。这和"观测优先"是一回事，但放在取舍这一节，是想强调它也是一个**人手层面**的决定：配额引擎要写 gate、要处理超额、要做退款边界，是一大块活；先不做，把这块人力省下来投到更要紧的地方（比如运行时统一）。

这三个取舍的共同逻辑是：**小团队最稀缺的不是技术，是注意力。** 每多一层复杂度，就多一处要持续维护、会出 bug、要占脑子的地方。在能接受的范围内主动少做一层，是把注意力留给真正要紧的事——这本身就是一种架构能力。

## 六、下一步（自我提醒，不是承诺）

不是路线图，是一份写给自己看的"已知该还的债"清单：

- **统一 per-message / per-session 运行时（Phase C）**：把两套运行时合并，理清工作区文件在 worker 和预览容器间的共享边界。这是当前最大的架构张力。
- **会话事件加 seq + 去重**：给前端 store 一把统一排序键，根治 resume 卡队列、消息重影。
- **持久化翻转成 DB 为真相 + `message` 表**：把消息的权威副本从 SDK 的 transcript 收回到自己的库里。
- **计费从观测进到配额**：换算率校准完，把 gate 接进生产流。
- **Skills 改 DB 目录 + 懒加载**：终结 copy-on-enable 的复制爆炸。
- **想清楚 SDK 钉版的退路**：把"迁移到原生 Anthropic + native binary"从"悬在头上"变成"有具体路径"。

## 反直觉结论

> [!IMPORTANT]
> **"基于官方 SDK 造产品"省掉的是内核，欠下的是耦合。**
>
> OxyGenie 最对的决策（包 SDK）和最大的几笔债（钉死 0.2.112、transcript 当真相、structured outputs 漏指令）是**同一枚硬币的两面**——你站在 SDK 上，就同时继承了它的能力和它的脾气。包住 SDK 让你不必重写 Loop、能跟随它演进，这是省下的内核；但你也因此把"版本怎么走、真相存在哪、特性会不会漏内部细节"这几个根基，部分交到了上游手里，这是欠下的耦合。**能力和耦合是同一个动作的两面，没有只要前者不要后者的办法。**

所以，真正的工程成熟度，不在于"用不用 SDK"——那是个被过度讨论的假问题。**成熟度在于：清楚地知道自己在哪些点上把命运交给了 SDK，并为每一个点准备退路。** 钉死 0.2.112，退路是想清楚原生 Anthropic 的迁移路径；transcript 当真相，退路是翻转成 DB 为真相；structured outputs 会漏指令，退路是退回启发式 + manifest。**每一个"交出去的点"，都配一个"拿回来的预案"——这才是站在巨人肩膀上还能站稳的姿势。** 这 19 篇，本质上就是把这些"交出去的点"一个个标记出来、再给每个标上退路。它不是一份成功学，是一张诚实的耦合地图——而一张诚实的地图，比一个漂亮的架构图有用得多。
